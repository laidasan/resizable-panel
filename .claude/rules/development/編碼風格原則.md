# 編碼風格原則

基於使用者偏好的編碼風格，撰寫程式碼時遵循以下原則。

## 單一出口原則

- 一個 function 只有一個回傳點
- 用變數（如 `result`）收集結果，流程走完後統一 `return`
- 不使用 early return 中斷流程
- 單步驟直接 return，不需要額外的 `result` 變數
  - Good: `return this._applyConstraints(layout, panels)`
  - Bad: `const result = this._applyConstraints(layout, panels); return result`

## 語意化封裝

- 型別判斷不直接使用原生 operator / API（如 `typeof`、`Number.isNaN()`），改用語意化的 utility function（如 `isNumber()`、`isNotNumber()`）
- 常數值不使用字串字面值，改用 Enum 或常數物件（如 `CssUnit.Percent` 取代 `'percent'`）
- 即使只使用一次，邏輯中的字串字面值也應賦予語意化名稱，消除魔術字串
- Enum / 常數物件獨立成檔案，一個 Enum 一個檔案
- 模組內部的映射表、查找表用 module level `const`，不放 class static
- 可支援的選項以 class static property 宣告，不硬編碼在邏輯中
- 變數名稱需語意化，讓讀者從名稱就能理解用途

### 變數命名判斷基準

以下情況代表名稱不夠語意化，需要改善：

1. **離開所在 function 就讀不懂** — 名稱需要依賴周圍程式碼才能理解用途
   - Bad: `each`（每個什麼？）→ Good: `remainingEachPercent`
   - Bad: `withDefault`（有什麼 default？）→ Good: `defaultSizePanels`
2. **名稱可能產生歧義** — 讀者可能誤解為其他概念或型別
   - Bad: `overflow`（像 boolean 或 CSS 概念）→ Good: `overflowSize`（明確是數值量）
3. **配對變數風格不一致** — 同一組對應變數的命名慣例應統一，除非資料結構本身不同
   - `defaultSizePanels`（`{ panel, percent }[]`）與 `nonDefaultPanelDataList`（`PanelData[]`）命名不對稱，但反映了實際的型別差異，屬合理情況

## 邏輯拆分

- 驗證、轉換等可獨立語意的邏輯，抽為 private method
- method 內部流程保持線性可讀，避免巢狀過深

## 純函式原則

- 函式不修改傳入的參數，需要變更時回傳新物件
- 內部輔助函式也遵守此原則，不透過參數做 in-place mutation
- 與「單一出口原則」互補：單一出口管流程結構，純函式管資料不可變性

## 變數宣告偏好（const 優先）

- 能用 `const` 就不用 `let`
- 多步驟轉換時，用各步驟獨立命名的 `const`，而非同一個 `let` 變數重新賦值
  - Good: `const rawLayout = ...` → `const normalizedLayout = ...` → `const result = ...`
  - Bad: `let layout = ...` → `layout = normalize(layout)` → `layout = clamp(layout)`
- 理由：`let` 會讓讀者誤以為函式有副作用或狀態變化

## Functional 優先順序

- 有直接對應的 functional 操作時（`map`、`all`、`partition`、`fromPairs`），優先使用
- 需要複雜 accumulator、反向遍歷 + 提前終止等場景，失去可讀性時，保留命令式寫法
- 判斷基準：functional 寫法是否讓意圖和語意更明確，而非盲目追求 functional

## 工具函式

- 優先使用 ramda 等 functional utility library 處理常見操作（如 `isNil`、`defaultTo`、`map`、`partition`）
- 專案層級的共用判斷（如型別檢查）集中在 `tools/` 目錄下
