# Task 文件規範

## 檔案命名

`<序號>-<名稱>.md`，序號以兩位數零補齊。子任務用 `-` 延伸（如 `06-1-重構-validateLayout-logic.md`）。

## 必要結構

每份 Task 文件必須包含以下三個部分，順序固定：

### 1. Metadata（blockquote）

```markdown
> 狀態：<狀態值>
> 依賴：<依賴的 Task 或「無」>
```

狀態值統一使用：`pending` / `in-progress` / `done` / `backlog`。

可選欄位視需要補充，常見的有：

| 欄位 | 適用情境 |
|------|---------|
| `對應 SA 章節` | 新模組開發，對應 SA 文件的章節 |
| `決策文件` | 重構/修正，有獨立的問題分析或決策文件 |
| `參考` | 功能擴充，參考的外部實作或文件 |
| `SPEC 修訂` | 變更影響到既有 SPEC 的章節 |

### 2. 驗收條件

以 bullet list 列出可客觀判斷的完成標準。每一條應該能回答「是或否」。

### 3. TODO（任務細項拆分）

以 markdown checkbox（`- [ ]` / `- [x]`）列出具體工作項目。

組織方式依任務性質選擇：
- **按方法/功能拆** — 適合新模組開發，每個 public method 一組
- **按 Phase 拆** — 適合跨模組的功能擴充，依實作階段分段
- **按變更類型拆** — 適合重構，分為新增/修改/移除

## Context 補充（可選）

Metadata 與驗收條件之間，依任務性質補充必要的背景資訊。沒有固定的 section 名稱，以下為常見模式供參考：

| 任務性質 | 常見補充 |
|---------|---------|
| 新模組開發 | 模組描述、Public API 表、資料結構 |
| 重構 / 修正 | 目標、變更範圍（影響檔案、新增/修改/移除方法）、新舊行為對照 |
| 功能擴充 | 目標、執行流程變化、設計決策 |
| Vue SFC | 元件結構（props / data / computed）、DOM 結構 |

原則：補充的資訊是為了讓讀者理解「這個 task 要做什麼、為什麼這樣做」，不需要的就不寫。

## 標題格式

```markdown
# Task <序號>: <名稱>
```

統一使用 `:` 作為分隔符。
